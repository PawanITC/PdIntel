What Needs to Be Created in Stripe

1. Products (4)

  • PDIntel Freemium — free tier · 1 council · used for trial reference flows
  • PDIntel Starter — £45/mo · up to 3 councils
  • PDIntel Growth — £90/mo · up to 10 councils
  • PDIntel Pro/Enterprise — custom pricing · unlimited councils
  • Each product carries metadata.planType so webhooks don't string-match names

2. Prices (4–6 recurring)

  • Starter Monthly — £45.00 GBP · interval: month · lookup_key: starter_monthly
  • Growth Monthly — £90.00 GBP · interval: month · lookup_key: growth_monthly
  • Starter Annual — £486.00 GBP · 10% off × 12 · create now, activate later
  • Growth Annual — £972.00 GBP · same strategy
  • Enterprise Custom — created per deal at provisioning time via API · amount varies per customer
  • Prices are immutable — use lookup_key in code, not the price ID directly

3. Tax Rate — UK VAT 20%

  • percentage: 20.0 · inclusive: false (added on top) · country: GB
  • Attached via automatic_tax on CheckoutSession and Subscription
  • Each new jurisdiction (EU, Ireland) needs its own Tax Rate object

4. Customer Portal Config (1)

  • Allow: plan upgrades · scheduled downgrades · cancellation at period end · payment method updates · invoice history
  • Backend generates a short-lived portal session URL via billingPortal.sessions.create()
  • Stripe hosts the entire portal — zero UI build required
  • PDIntel branding: logo + brand colour applied in Dashboard once

5. Webhook Endpoint (1 URL · 10 events)

  • https://api.pdintel.com/webhooks/stripe registered in Stripe Dashboard
  • checkout.session.completed — new subscription purchased · trigger provisioning
  • customer.subscription.created — enterprise direct-create path
  • customer.subscription.updated — plan change · proration · schedule phase
  • customer.subscription.deleted — cancellation effective · revoke access
  • invoice.payment_succeeded — renewal paid · extend subscription period
  • invoice.payment_failed — start dunning flow
  • invoice.finalized — invoice locked · record in DB
  • setup_intent.succeeded — enterprise card vaulted · trigger subscription create
  • customer.subscription.trial_will_end — 3-day warning · send reminder email
  • payment_intent.payment_failed — card declined at checkout · notify user
  • Webhook Signing Secret (whsec_…) stored in Secrets Manager — never in code

6. Coupons (on demand)

  • Created per enterprise deal — not pre-created in bulk
  • Percent off — e.g. 20% off · duration: repeating · duration_in_months: 12
  • Fixed amount — e.g. £500 off · amount_off: 50000 · duration: once
  • Attached directly to subscriptions.create(coupon: id) — no Quote needed
  • metadata: { dealId, tenantId } for full audit trail

7. SetupIntent (enterprise path)

  • Generated per deal at CSM provisioning time — not pre-created
  • usage: off_session — triggers stronger 3DS upfront so future charges don't fail
  • customer: stripe_customer_id — vaults payment method against the tenant
  • metadata: { tenantId, dealId, provisionedBy: csmEmail }

System Architecture — EKS &

Ingress Security Chain

  • CloudFront: Global CDN · TLS 1.3 enforced · HTTPS-only · DDoS absorption
  • AWS WAF: OWASP Top 10 rules · IP rate-limiting · geo-blocking · SQL injection / XSS filters
  • AWS Cognito: User Pools · OAuth2/OIDC login · issues JWT (RS256) · federated IdP support · Lambda triggers for custom auth
  • API Gateway: JWT authoriser validates every request · per-route throttling · usage plans · request schema validation

Backend Services

  • Auth Service: RBAC roles · BILLING_WRITE / BILLING_READ / ADMIN
  • Billing Service: All API calls — sessions, webhooks, quotes
  • Subscription Service: Plan lifecycle · Kafka consumer · council mgmt
  • Entitlement Service: Feature gates · Redis cache · <5ms p95
  • Notification: SES email · SNS push

Design Principle

No service accesses another service's database directly. All cross-service data flows via REST (sync reads) or Kafka (async writes). Enables independent scaling and deployment.

Happy Flow — New Subscription (Starter / Growth)

Customer Find-or-Create

  • First purchase: customers.create() called once — Stripe Customer ID stored in tenants.stripe_customer_id
  • Repeat purchase: customers.retrieve(cus_xxx) — reuse existing Customer so all invoices, payment methods, and history stay linked
  • Freemium tenants: Customer created lazily — only at first paid checkout, not at signup
  • Key params: email, name, metadata.tenantId — needed to trace any Stripe event back to a tenant
  • Never duplicate: always check DB first — duplicate customers break invoice history and reporting

checkout.sessions.create Params

  • customer: cus_xxx (from find-or-create above)
  • mode: 'subscription'
  • line_items[].price: price_starter_monthly or growth_monthly (lookup_key)
  • line_items[].quantity: council count (e.g. 3 or 8)
  • metadata: tenantId, planType, councilIds CSV
  • automatic_tax: enabled — 20% UK VAT applied
  • success_url / cancel_url: frontend redirect targets
  • idempotency key: checkout_{tenantId}_plan_epoch

Kafka Event Flow

  • Topic: billing.events
  • Key: stripe_event_id (partition order)
  • Consumer: subscription-service group
  • Failure: DLQ after 3 retries
  • Internal: subscription.activated fires entitlement + notification

Upgrade Flow — Prorated Billing

Example: Basic (3 councils) → Growth (8 councils) on day 15 of 30

Proration Rules

  • Timing: Charged the moment the API call succeeds — not at next cycle
  • Credit: Unused days on old plan automatically credited by
  • Billing mode: CREATE_PRORATIONS (Stripe default)
  • Invoice: Issued immediately with line-item breakdown
  • Access: Granted on webhook — typically within seconds

Mid-Cycle Council Add

  • Same proration logic applies for adding councils without changing plan
  • Endpoint: POST /subscriptions/{id}/add-councils
  • quantity updated: e.g. 3 → 5
  • Prorated charge for added councils only
  • New council_subscription rows inserted immediately
7

Downgrade Flow — Scheduled, No Refund

Example: Growth (8 councils) → Basic (3 councils) — effective at next billing cycle end

Downgrade Rules

  • No refund: Billing amount unchanged for current cycle
  • Scheduled change: Takes effect on next renewal date only
  • Full access: Growth entitlements fully retained until period end
  • Council selection: Top N by creation date are kept; extras deactivated
  • Cancellable: Customer can cancel the downgrade before it takes effect

Council Removal (Same Logic)

  • DELETE /subscriptions/{id}/councils/{councilId}
  • Status set to PENDING_REMOVAL — access retained now
  • qty reduced at next billing renewal
  • No refund for current cycle
  • Customer notified 3 days before removal takes effect

Pro / Enterprise Flow — Quotes + Payment Links

Sales-assisted · Custom pricing · Unlimited councils · SubscriptionSchedule for renewals

Objects Used

  • Quote: Custom line items, discount, legal footer, PDF
  • Coupon: percent_off or amount_off, one-time or recurring
  • Payment Link: From accepted Quote, configurable expiry
  • SubscriptionSchedule: Multi-phase — current + renewal terms
  • Customer: Created once, reused for all future invoices

Why Not Checkout Session?

  • Enterprise pricing requires negotiation — not a web form
  • Legal terms, custom SLAs, and discount approvals need review
  • Sales team controls Quote issuance, not the customer
  • SubscriptionSchedule preserves agreed price across multiple renewal years

Enterprise Alternate Flow — SetupIntent + Admin Portal

Phase 1 — Deal Trigger

  • HubSpot webhook fires on "Deal Closed Won" stage — calls POST /admin/enterprise/provision automatically
  • No manual handoff — sales closing the deal is the trigger
  • CSM can also submit manually from Admin Portal (ADMIN role)

Phase 2 — Customer + Price

  • customers.create() — one-time · stores cus_xxx in tenants table for all future calls
  • prices.create() — custom unit_amount per deal · recurring monthly · lookup_key: enterprise_{tenantId}
  • Both Stripe IDs stored in DB — no re-creation on retry

Phase 3 — SetupIntent

  • usage: off_session — forces strong 3DS auth upfront so future server-side charges never get declined
  • Stripe hosts the card entry page — PDIntel never touches card data
  • CSM emails the URL — no frontend build needed
  • Webhook setup_intent.succeeded is the trigger for subscription creation

Phase 4 — Subscription

  • billing_cycle_anchor — exact start date from contract, not "now"
  • trial_end — optional grace period before first charge
  • coupon — discount attached directly, no Quote needed
  • SubscriptionSchedule created immediately after for multi-year phase management
  • Access provisioned only after customer.subscription.created webhook — same pattern as self-serve

Enterprise Flow — Current vs Better Alternative

Current Approach — Quote → PaymentLink
Who initiates
  • Sales team inside Stripe Dashboard — backend is a passive observer
  • Backend only wakes up when a webhook arrives after payment
Flow (10 steps)
  • Quote created → Coupon attached → PDF emailed → Customer accepts → PaymentLink generated → Customer pays → webhook → provision
Stripe objects used (7)
  • Customer · Price · Quote · Coupon · PaymentLink · Subscription · SubscriptionSchedule
Pros
  • Faster to ship — mostly Stripe Dashboard + webhooks, minimal backend code
  • Stripe-generated Quote PDF is a legally referenceable document version-tracked inside Stripe
  • Zero frontend build — Stripe hosts PaymentLink page
  • Discounts visible on invoice via Stripe Coupon
Cons
  • Sales team needs Stripe Dashboard access + training
  • If PaymentLink fails or customer never pays, state is stuck in Stripe — hard to retry programmatically
  • CRM integration is manual — deal stages must be mirrored by hand
  • Subscription start date is controlled by when customer clicks Pay — not by your backend
  • 7 Stripe objects = more failure points across the chain
Recommended — SetupIntent + Admin Portal
Who initiates
  • CSM submits a provisioning form in the PDIntel Admin Portal (ADMIN role)
  • Backend is the initiator — calls Stripe directly, no Dashboard involvement
Flow (6 steps)
  • POST /admin/enterprise/provision → Customer + Price + SetupIntent created → CSM emails Stripe-hosted setup URL → Customer vaults card → setup_intent.succeeded webhook → subscriptions.create() with exact billing_cycle_anchor → SubscriptionSchedule for renewals
Stripe objects used (5)
  • Customer · Price · SetupIntent · Subscription · SubscriptionSchedule
Pros
  • Backend has full control — subscription start date, trial end, billing anchor set precisely
  • Failure recovery is in your own DB — retry is a single API call, no Stripe Dashboard state to unpick
  • Natural CRM hook — HubSpot "Deal Closed" webhook can POST /admin/enterprise/provision directly
  • No Stripe Dashboard training for sales team — everything through admin portal
  • Mid-contract changes (seat add, trial extension, amendment) handled by backend, not Stripe UI
Cons
  • Requires building the Admin Portal provisioning form — additional frontend work
  • No Stripe-generated contract PDF — paper trail lives in DocuSign / email (external to Stripe)
  • Audit trail for deal terms must be maintained in your own DB
Verdict — Recommended for PDIntel
  • Use SetupIntent + Admin Portal for enterprise provisioning — backend control pays off as deal complexity grows
  • Keep DocuSign / email for the contract paper trail — Stripe is a payment rail, not a CRM
  • Stripe Coupon still attaches directly to subscriptions.create() — no Quote needed for discounts
  • SubscriptionSchedule is identical in both approaches — multi-year pricing preserved either way
  • CRM trigger (HubSpot → POST /admin/enterprise/provision) eliminates manual sales handoff entirely

Deactivation / Cancellation Flow

Cancels at period end · Access retained · Data held 90 days · Dunning on payment failure

Cancellation Rules

  • No immediate cutoff: Access held until billing cycle end
  • No refund: Remaining days not refunded
  • Re-subscribe: Customer can reactivate before expiry to avoid gap
  • Data: Retained 90 days, then purged per GDPR
  • Stripe Customer: Kept for potential future re-subscribe

Dunning & Grace Period

  • Dunning triggered by: invoice.payment_failed webhook
  • 3 automatic retries via Smart Retries
  • Each failure: customer notified by email via SES
  • 7-day grace: subscription stays ACTIVE but flagged PAST_DUE
  • After grace: subscription auto-cancelled, all access revoked

Cognito

  • User Pools: Manages user identities — sign-up, sign-in, MFA
  • OAuth2 / OIDC: Issues JWT access + ID tokens on successful auth
  • Federated IdPs: Google, Microsoft, SAML for enterprise SSO
  • Token expiry: Access token 15 min · Refresh token 7 days
  • App Clients: Separate clients for web and mobile with scopes
  • Triggers: Lambda pre-token hooks to inject tenant claims into JWT

API Gateway

  • JWT Authoriser: Validates Cognito-issued tokens on every request
  • Throttling: 100 req/s per route (auth), 1 000 req/s (data)
  • Usage plans: Per-tenant API keys for metered access control
  • Route mapping: /auth/* → Auth Service · /billing/* → Billing Service
  • Request validation: Schema checks before traffic hits EKS
  • Access logs: All requests logged to CloudWatch

Network Controls

  • TLS 1.3: Enforced on CloudFront + API Gateway — 1.0/1.1 disabled
  • WAF rules: Geo-blocking, SQLi, XSS, rate-limit per IP
  • CORS: Allow-list: dev.pdintel.co.uk + app.pdintel.co.uk only
  • Secrets Manager: Credentials fetched at startup, rotated every 30 days
  • KMS: Encryption at rest — RDS, ElastiCache, MSK, S3
  • Private subnets: EKS, RDS, Redis — not internet-routable

Webhook Security

  • HMAC-SHA256: Stripe-Signature verified before any processing
  • Replay guard: Timestamp tolerance ±300 seconds
  • Webhook secret: Stored in Secrets Manager — never in code
  • Idempotency: Redis SETNX per stripe_event_id (24h TTL)
  • Immediate 200: Returned before processing — prevents retry storms
  • DLQ: Events failing 3 retries go to dead-letter queue

Compliance, Data Protection & Observability

PCI-DSS (SAQ-A)

  • Hosted Checkout: Card input rendered in Stripe iframe — keystrokes never reach PDIntel JS
  • Zero card storage: No card numbers, CVVs, or magnetic data on PDIntel servers
  • Minimal PCI scope: SAQ-A qualifies because all card data handled by
  • Stripe.js: Loaded from js.stripe.com — PCI-certified CDN
  • Tokenisation: tokenises card before any backend call is made

UK GDPR

  • Data residency: All PII stored in eu-west-2 (London)
  • Data minimisation: Only collect what is needed for billing and access
  • Right to erasure: DELETE /tenants/{id} anonymises PII; data purged after 90-day retention
  • Right to access: GET /tenants/{id}/data export available to all users
  • DPA with Stripe: Data Processing Agreement in place — acts as data processor
  • Audit trail: audit_events table is append-only — 7-year retention (HMRC)
